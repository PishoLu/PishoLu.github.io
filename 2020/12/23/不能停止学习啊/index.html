setState 和 useState

setState 是异步还是同步

由于公司用的是 Hook 函数式组件，所以 setState 方面都忘得差不多了，这两天重新回顾了一下。
首先，setState 和 useState 都是 React 保存状态的方法，一个是类组件通过添加 state 属性，通过 this.setState 调用，一个是函数式组件中通过 useState 方法，会生成一个 state 对象和一个 set 的方法。
他们两个的作用是一样的，都是去更新组件的 state 状态，但是让人奇怪的是他们的使用不一样。
setState 接收两个参数，一个是 state 新的值，一个是一个回调函数。
useState 返回回来的 set 方法只接收一个参数，或者是最新值，或者是一个函数，这个函数的参数是当前的 state 值。

从上面两种不同的更新 state 的方法中，我们可以发现，他们的原理并不是相同的。说明 React 在内部细节上是有区别的，让我们回顾一下这个面试题，setState 是异步还是同步的。

首先我们判断他是一个同步的代码，因为异步的是一个 Promise 或者其他的异步类型，可以后续.then()继续逻辑，但是这里的 setState 却是通过一个回调函数来实现后续的处理。

那么如果他是一个同步的方法，为什么在外面拿不到最新值呢，只能通过回调函数去获取最新值。这个就跟 React 的处理机制有关系了。

在一个复杂应用网页中，state 会有很多，他们的修改也会很多，如果每一次修改都去重新渲染一次的话就会非常影响页面性能。所以 React 在处理 state 更新的时候会等上一会儿，如果有其他的 state 或者同一个 state 的多次更新，就可以一起更新然后只渲染一次，提升性能。

而在函数式编程中，这个同步的方法就更加明显了，要不是同样也拿不到最新值，我都怀疑 React 没有做处理了。因为相比于用 useEffect 去监听这个 state，用一个回调函数去处理后续的操作，代码的逻辑会更舒服一点。

所以这个题其实就是个陷阱题，同步异步都可以，只要你能说出来 React 更新机制就行

Redux

让我先介绍一下 Redux，然后问我用没用过一些中间件。

关于 Redux 之前是看过文档的，不过现在都忘得差不多了。可以参考 React 学习

说起来一直在用 Dva，但是发现自己连 Dva 的文档也没认真看过，真是废物。

在 Redux 中的 Action 中，是不能使用异步操作的，在这种情况下，就可以使用第三方的中间件，而在 Dva 中，可以通过 effects 去执行异步操作，于是就有了一些中间件来帮助 Redux 获取这些能力，比如 Redux-thunk Redux-Promise

关于 Thunk 在 JS 的笔记中也有，不过在这个插件的 github 上有个很有意思的代码。

可以再看看阮一峰的Thunk 函数的含义和用法

之前看的Redux 的文档确实是有中间件的介绍的，后悔没有仔细看，当时我听到中间件都听不懂。

  • redux-thunk — 用最简单的方式搭建异步 action 构造器
  • redux-promise — 遵从 FSA 标准的 promise 中间件
  • redux-axios-middleware — 使用 axios HTTP 客户端获取数据的 Redux 中间件
  • redux-observable — Redux 的 RxJS 中间件
  • redux-rx — 给 Redux 用的 RxJS 工具，包括观察变量的中间件
  • redux-logger — 记录所有 Redux action 和下一次 state 的日志
  • redux-immutable-state-invariant — 开发中的状态变更提醒
  • redux-unhandled-action — 开发过程中，若 Action 未使 State 发生变化则发出警告
  • redux-analytics — Redux middleware 分析
  • redux-gen — Redux middleware 生成器
  • redux-saga — Redux 应用的另一种副作用 model
  • redux-action-tree — Redux 的可组合性 Cerebral-style 信号
  • apollo-client — 针对 GraphQL 服务器及基于 Redux 的 UI 框架的缓存客户端
