对 Angular 的理解

发表于 2021-03-20 分类于 学习
Angular 和 React 或者 Vue 感觉是两个东西一样。由于面向对象的设计模式，复杂的文档和 Api 导致入门比较难。但是实际上用起来还是挺不错的。

对 Angular 的理解和基本学习

对于’框架们’的理解

不管是 Angular、React 还是 Vue，这些 Web 框架的出现，都是为了让开发者更加方便、快捷的开发 Web 页面。在前端技术发展的这十多年，这三个框架成为了最后的幸存者，自然是有他们的技术优势在的。

不论是什么框架，都可以将他们的功能概括为

  • 组件

    包含页面、功能性函数、状态等功能，是开发过程中的最小单位。

  • 模版语言

    精简页面逻辑，减少代码量。各个框架的形式可能不太一样，但是原理基本一样。Jsx 真是个好东西，写起来太舒服了。

  • 响应式原理

    不管是单向绑定还是双向绑定，都是为了将值准确的更新在页面视图上。

  • 路由

    不论是大型项目还是小型项目，多页应用还是单页应用，路由都是必不可少的。即使 Vue 和 React 为了减少包大小将 Router 功能 单独拿了出来，但是基本上每个项目都会去装。

所以在学习 React 和 Vue 的时候，其实学起来是很快的，因为这些内容基本上是差不多的。学会了一个，很快就能学会另一个。React 和 Vue 最大的区别无非是模版语言的形式不一样，以及生命周期名字不一样，对于一个普通的开发者来说，区分和相互学习是很容易的一件事。

Angular 框架在国内的应用并不是很多。但是这并不说明 Angular 比其他两个差到哪里去。由于 Angular 较快的更新迭代，不太了解之前的版本，所以从现在的 11 版本说起。

Angular 最大的区别和学习起来的成本比其他两个框架更高的地方就在于 Angular11 版本高度支持 TypeScript，再加上其面向对象的装饰器模式，框架本身集成了很多的功能，导致你需要用很长的时间去学习和熟悉不同的装饰器、类的功能。

Angular 的好处在于，由于出色的设计模式，使得这套框架非常适合一些大型应用，而且集成了 TypeScript 之后开发过程中的代码逻辑更加完善。

Angular 的项目结构

一个项目跑起来，找到入口，就可以一层一层的去剖析代码的运行过程。这也是最快上手一个项目的方法。

Angular 官方的生态是真的很不错，该有的东西基本上都有了，第三方的框架都是为其添砖加瓦，而不是以此为基础去盖楼。通过 Angular-cli 命令行工具生成项目。

      • styles.less

由于 Js 导入导出的特性，所以不用太在意文件本身的结构。而是要去看代码之间的关联。

首先看的肯定是这个明显的入口文件 main.ts。省略掉前面的引入，它使用了一个 platformBrowserDynamic() 的函数，这个函数是从 @angular/platform-browser-dynamic 库中导出的。查询了官方文档后我们知道这个函数是用来即时编译代码并支持各类浏览器用的，也就是进行 Web 开发时的入口函数，出口当然就是生成 html 文件给浏览器了。该函数返回了一个 PlatformRef 接了一个 bootstrapModule() 的函数，用来生成引导模块。引导模块相比就是函数里的 AppModule 了。然后我们去看看这个引导模块。

先找到主体，是下面被 export 出去的 AppModule 类，这个类被一个 @NgModule 的装饰器装饰，这个也是 Angular 的一个关键装饰器，这个装饰器可以接受一个对象作为参数，查询文档后我们了解到，这个装饰器的参数的意义。

  • providers?

    在当前模块的注入器中可用的一组可注入对象。

  • declarations?

    属于该模块的一组组件、指令和管道（统称可声明对象）。

  • imports?

    这里列出的 NgModule 所导出的可声明对象可用在当前模块内的模板中。

  • exports?

    此 NgModule 中声明的一组组件、指令和管道可以在导入了本模块的模块下任何组件的模板中使用。 导出的这些可声明对象就是该模块的公共 API。

  • entryComponents?

    定义此 NgModule 中要编译的组件集，这样它们才可以动态加载到视图中。

  • bootstrap?

    当该模块引导时需要进行引导的组件。列在这里的所有组件都会自动添加到 entryComponents 中。

  • schemas?

    该 NgModule 中允许使用的声明元素的 schema（HTML 架构）。 元素和属性（无论是 Angular 组件还是指令）都必须声明在 schema 中。

  • id?

    当前 NgModule 在 getModuleFactory 中的名字或唯一标识符。 如果为 undefined，则该模块不会被注册进 getModuleFactory 中。

  • jit?

    如果存在，则该指令/组件将被 AOT 编译器忽略。它会保留在发布代码中，并且 JIT 编译器会尝试在运行时在浏览器中对其进行编译。为了确保其行为正确，该应用程序必须导入 @angular/compiler 。

先来看引入的这两个模块：

  • BrowserModule

    导出了所有 Angular 应用都需要的基础设施。二次导出了 CommonModule 和 ApplicationModule，即 Angular 模版语言中的指令、管道等功能由这个模块完成，以及 ApplicationModule 表示将该模块作为一个应用来看。

  • AppRoutingModule

    是从同级目录下引入的。细看。

这个也是一个模块，引入 RouterModule 的 forRoot() 函数返回回来的包含路由信息的组件模块。再将 RouterModule 导出。也就是说这是一个路由模块，用来生成路由信息和绑定路由到组件。

它被引入到 AppModule 中即 AppModule 包含了这个路由下的所有组件。但是这个路由里面没有包含任何东西。那么我们访问的什么呢？访问到的是 AppComponent 这个组件，这个组件在 AppModule 中被引入到 declarations 中，所以其实我们可以将他移到 AppRoutingModule 中来，我觉得可以更加容易理解一些。

那么这个 AppComponent 一定就是一个页面组件了。

一个组件，通过 @Component 装饰器装饰，包含的参数通过文档去看。常用的：

  • selector

    告诉 Angular 怎样在 html 模版中定位到该组件。

  • templateUrl

    实现该组件的模版

  • styleUrls

    模版中使用的样式

所以整个文档的结构差不多是这样

  • 根模块或者说应用
    • 模块
      • 路由模块？
      • 组件
        • 组件类
        • 模版？
    • … 更多的模块

即，Angular 将根模块作为整个页面应用的入口，再向下包含更多的模块，以每个模块作为解析单位，通过模块使用的或引入的路由模块，去加载不同的组件。每个组件一定会被模块引入，一个模块可以引入多个组件，包括指令、管道等功能性组件。不一定会有具体的模版。

所以在掌握了 @NgModule @Component 这两个基础的装饰类的用法后，就可以开始开发具体的页面了。模版中的插值、绑定等语法什么的实在是熟悉的不能再熟悉了。不过也还是被 Angular 玩出了一点花样。

模版语法

为了方便上手和开发，框架们都不约而同的选择了 html 作为模版语言的承载体，通过 jsx，v-指令和 ng-指令等方式精简代码逻辑，强化 html 的动态能力。

如果要我说的话，我最喜欢当然还是 jsx 了。非常自由，也符合 js 的编程习惯。Vue 的指令模式也不是不能接受。Angular 就显得很奇怪，明明大家都是指令模式，Angular 非要区分成属性型指令和结构性指令，感觉没有这个必要。简单来说的话，Angular 的指令分为两种，使用 [] 包裹的，和使用 * 作为前缀的。使用 * 的指令其实就是一种语法糖，由于这类指令通常与 DOM 结构相关，例如 ngIf ngFor 等，* 语法糖会将其处理成通过 <ng-template> 标签包裹的形式。详情参考。

然后就和其他的模版语言没什么区别了。使用 @Componet 的类作为模版的上下文，去使用变量、函数、生命周期等方法去动态控制 html 模版。
